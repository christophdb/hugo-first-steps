Здесь представлен неполный список возможных проблем, ошибок или часто задаваемых вопросов по Single Sign-On для SeaTable Cloud.

{{< faq "Можно ли использовать несколько доменов электронной почты в одной команде?" >}}
Участники команды могут использовать разные домены электронной почты в SeaTable. Однако после подключения вашего поставщика идентификации (IdP) к команде SeaTable к одному IdP может быть привязан только один домен.
{{< /faq >}}

{{< faq "Можно ли указать несколько IdP для одной команды SeaTable?" >}}
В настоящее время невозможно настроить более одного поставщика идентификации (IdP) для одной команды. На каждую команду разрешается только один IdP.
{{< /faq >}}

{{< faq "Какие атрибуты SAML должен передавать поставщик идентификации?" >}}
Для успешного использования Single Sign-On в SeaTable Cloud поставщик идентификации (IdP) должен передать лишь три атрибута пользователя:

- `uid` (уникальный идентификатор пользователя, который остается неизменным на протяжении всего жизненного цикла пользователя)
- `contact_email` (адрес электронной почты пользователя)
- `name` (полное имя пользователя)

Другие атрибуты являются необязательными и игнорируются SeaTable.
{{< /faq >}}

{{< faq "Что происходит, если пользователь уже является членом другой команды?" >}}

Пользователь может принадлежать только к одной команде. Если пользователь пытается войти через Single Sign-On (SSO) и уже является членом другой команды, будет отображено соответствующее сообщение об ошибке. Вход в систему в таком случае невозможен.

{{< /faq >}}

{{< faq "Что происходит, если моя команда уже заполнена?" >}}

Если новый пользователь, не зарегистрированный ранее, пытается войти в вашу команду через Single Sign-On, он автоматически добавляется в команду, если остаются свободные места. Если максимальное количество участников команды достигнуто, новые пользователи через SSO войти не смогут. Вход будет неудачным.

{{< /faq >}}

{{< faq "Работает ли вход через Single Sign-On, если пользователь был деактивирован?" >}}

Если пользователь был деактивирован в управлении командой, он немедленно выходит из SeaTable Cloud и больше не может войти через Single Sign-On.

{{< /faq >}}
